## Summary

Mike and Matthias catch up on fast-moving model news, including Fable’s disappearance and Anthropic’s paused plan to restrict subscription-based headless usage. They discuss emerging alternatives like GLM 5.2 and why provider-level differences in pricing, API support, and data retention policies matter in production. The main segment explains “loop engineering” as the next step after prompt and context engineering: designing agent systems that generate prompts, verify outputs, and re-run work automatically. They close with practical multi-agent patterns using GitHub issues and Actions, plus advice on pairing cheaper implementer models with stronger reviewer models.

## Key Takeaways

- ▸ Anthropic reportedly paused its June headless-subscription restriction, which the hosts interpret as likely community-driven pushback.
- ▸ Data retention and zero-retention guarantees vary by provider even for the same model, so provider selection is a security decision, not just a cost decision.
- ▸ Loop engineering is defined as building systems where agents prompt other agents, rather than humans manually writing every next prompt.
- ▸ Robust verification loops are the core technical challenge; without trusted automated validation, higher-level delegation breaks down.
- ▸ A scalable architecture is a long-running “manager” agent that delegates implementation/review work to short-lived sub-agents with separate context windows.
- ▸ Cost efficiency can improve by using cheaper models for implementation loops and reserving premium models for planning/review and final judgment.
